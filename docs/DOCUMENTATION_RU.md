# ReStreamFox — Техническое руководство

## Архитектура системы

ReStreamFox — это микросервисный стек вещания, развернутый с использованием Docker Compose. Стек включает в себя четыре основных компонента:

```
                  ┌──────────────┐
                  │OBS / Источник│
                  └──────┬───────┘
                         │ (RTMP/SRT c авторизацией)
                         ▼
                  ┌──────────────┐
                  │   MediaMTX   │◄──────────────┐
                  └──────┬───────┘               │
                         │ (RTSP)                │ (Опрос API)
                         ▼                       │
   ┌──────────┐   ┌──────────────┐   ┌───────────┴──┐
   │  Адреса  ├──►│    FFmpeg    │◄──┤   Watchdog   │
   │  (.txt)  │   │ Транскодер   │   │ (Управление) │
   └──────────┘   └──────────────┘   └──────────────┘
                         │ (мультиплексор tee)
                         ▼
             [ YouTube / Twitch / Kick ]
```

1. **MediaMTX**: Принимает входящий поток от источника (OBS, vMix и т.д.) по протоколам RTMP, RTMPS, SRT, RTSP или WebRTC.
2. **Watchdog**: Легковесная служба на Python, опрашивающая API MediaMTX. При появлении активного потока переключает FFmpeg в режим трансляции, при отключении — активирует режим заставки.
3. **FFmpeg**: Кодирующее ядро. Контролируется через `entrypoint.sh`, который на лету реагирует на сигналы от watchdog и запускает соответствующий bash-скрипт (`transcode.sh` или `backup.sh`). Использует мультиплексор `tee` для раздачи потока на целевые платформы.
4. **UI**: Веб-интерфейс на Flask и Socket.IO. Позволяет отслеживать состояние системы в публичном режиме, а также предоставляет админ-панель (`/admin`) для просмотра логов в реальном времени и управления режимами.

## Параметры конфигурации (`config.yml`)

Файл конфигурации валидируется службой watchdog при запуске контейнеров.

| Раздел | Ключ | Тип | Описание |
| :--- | :--- | :--- | :--- |
| - | `mediamtx_api` | Строка | Базовый URL API MediaMTX (например, `http://mediamtx:9997`). |
| - | `path_name` | Строка | Имя пути входящего потока (по умолчанию `live`). |
| - | `check_interval` | Целое число | Интервал опроса API MediaMTX в секундах. |
| `backup` | `enabled` | Булево | Активировать ли режим заглушки при отсутствии потока. |
| `backup` | `file` | Строка | Путь к файлу заглушки: видео (`.mp4`) или картинка (`.png`/`.jpg`). |
| `ffmpeg.video` | `codec` | Строка | Кодек сжатия видео (например, `libx264`). |
| `ffmpeg.video` | `bitrate` | Строка | Целевой битрейт видео (например, `4500k`). |
| `ffmpeg.video` | `gop` | Целое число | Размер GOP для синхронизации кадров. |
| `ffmpeg.audio` | `codec` | Строка | Кодек сжатия звука (например, `aac`). |
| `ffmpeg.audio` | `bitrate` | Строка | Целевой битрейт звука (например, `128k`). |
| `ui` | `admin_password` | Строка | Пароль администратора для авторизации на `/admin`. |
| `ui` | `admin_token` | Строка | Токен доступа к REST-методам (передается в X-Admin-Token). |
| `ui` | `secret` | Строка | Секретный ключ для подписи сессий Flask. |

## Мультиплексирование протоколов

FFmpeg использует эффективный мультиплексор `tee`, распределяющий поток на уровне контейнера данных. Поддерживается гибкое определение типа протоколов:

* **RTMP / RTMPS**: Автоматически упаковывается в медиаконтейнер `flv` (требуется для YouTube, Twitch, Kick). TLS-шифрование для RTMPS поддерживается нативно.
* **SRT / UDP**: Автоматически упаковывается в медиаконтейнер `mpegts` для обеспечения устойчивого вещания при потерях сетевых пакетов.

## Безопасность входящего вещания (Ingest Auth)

Для предотвращения трансляции неавторизованных пользователей на ваш сервер, путь `live` защищен параметрами аутентификации.

Настройка осуществляется в `mediamtx/mediamtx.yml`:
```yaml
paths:
  live:
    publishUser: "streamer"
    publishPass: "secure_ingest_key_here"
```

При этом внутренний FFmpeg-транскодер забирает локальный RTSP-поток без пароля, так как параметры `publishUser` и `publishPass` защищают только действие публикации (`publish`), оставляя чтение (`read`) доступным для локальных служб сервера.

Для вещания из OBS Studio необходимо использовать следующий синтаксис:
* **Для SRT**: `srt://<IP>:8890?streamid=publish:live:streamer:secure_ingest_key_here`
* **Для RTMP**: `rtmp://<IP>:1935/live?user=streamer&pass=secure_ingest_key_here`

## Режимы резервного вещания

При переключении в режим заглушки скрипт автоматически определяет тип медиафайла:
- **Видеофайлы**: Циклически декодируются и отправляются на стриминг в бесконечном цикле.
- **Статичные изображения**: Декодируются как непрерывный видеоряд (25 кадров/с) с генерацией виртуальной аудиодорожки стерео-тишины (`anullsrc`). Это гарантирует корректную работу плееров на стороне платформ и предотвращает отключение трансляции за отсутствие звукового сопровождения.
